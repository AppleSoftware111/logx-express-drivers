━━━━━━━━━━━━━━━━━━━━━━━━━
MONGODB-SPECIFIC RULES
━━━━━━━━━━━━━━━━━━━━━━━━━

Query rules:
- Always filter by companyId first in every query (multi-tenant safety)
- Use .lean() on all read-only queries for performance
- Use .select() to never over-fetch fields
- Use .populate() sparingly — only what the frontend actually needs
- Use aggregation pipeline for reports, never load full collections

Duplicate execution prevention (CRITICAL):
  await RouteExecution.findOneAndUpdate(
    { routeId, scheduledDate },
    { $setOnInsert: { ...executionData } },
    { upsert: true, new: true }
  )
  This is atomic — safe even if cron runs twice

GPS bulk insert:
  await GpsPoint.insertMany(bufferedPoints, { ordered: false })
  Run every 30 seconds from in-memory buffer

Geofencing query (find executions near a point):
  await Client.findOne({
    location: {
      $near: {
        $geometry: { type: 'Point', coordinates: [lng, lat] },
        $maxDistance: 100 // meters
      }
    }
  })

Waiting time calculation:
  const waitingTimeMinutes = Math.round(
    (completedAt.getTime() - arrivedAt.getTime()) / 60000
  )
  Save directly to ExecutionStop.waitingTimeMinutes — never recompute on frontend

Reports aggregation example:
  RouteExecution.aggregate([
    { $match: { companyId, scheduledDate: { $gte: start, $lte: end } } },
    { $unwind: '$stops' },
    { $group: {
        _id: '$driverId',
        totalStops: { $sum: 1 },
        avgWaitingTime: { $avg: '$stops.waitingTimeMinutes' },
        completedStops: { $sum: { $cond: [{ $eq: ['$stops.status','COMPLETED'] }, 1, 0] } }
    }}
  ])
